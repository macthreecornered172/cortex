## Functional Requirements

- **FR1:** `TaskPush.push/3` sends a `TaskRequest` to an agent via the appropriate transport (gRPC stream pid or WebSocket channel pid), returning `{:ok, :sent}` or `{:error, reason}`.
- **FR2:** `Registry.route_task_result/2` replaces the current no-op with logic that looks up a pending task by `task_id` and delivers the result to the waiting caller via `PendingTasks.resolve_task/2`.
- **FR3:** `AgentChannel.handle_task_result/2` calls `Registry.route_task_result/2` with the validated task result (currently it only acks and drops the payload).
- **FR4:** Unknown `task_id` values in `route_task_result/2` are logged as warnings with telemetry, not silently dropped.
- **FR5:** Transport send failures (dead pid, closed stream) return `{:error, :transport_down}` from `TaskPush.push/3`.
- **Tests required:** Unit tests for `TaskPush` (gRPC path, WebSocket path, dead pid), unit test for updated `route_task_result/2`, integration test covering the full push-then-result loop.

## Non-Functional Requirements

- Language/runtime: Elixir 1.16+ / OTP 26+
- Observability: Emit telemetry events `[:cortex, :gateway, :task, :dispatched]` on push and `[:cortex, :gateway, :task, :completed]` on result delivery (events already defined in `Cortex.Telemetry`).
- Safety: `TaskPush.push/3` must handle dead pids gracefully — no crashes, no linked process failures. Use `Process.alive?/1` checks and try/rescue around send operations.
- Documentation: `@moduledoc`, `@doc`, `@spec` on all public functions per project conventions.
- Performance: No benchmarks needed — this is a control-plane path, not a hot data path. Message dispatch is O(1) per agent.

## Assumptions / System Model

- **Deployment environment:** Single-node Elixir application; Gateway, Registry, and Provider.External all run in the same BEAM VM.
- **Failure modes:**
  - gRPC stream pid dies before `TaskRequest` is delivered → `{:error, :transport_down}`.
  - WebSocket channel pid dies before push → `{:error, :transport_down}`.
  - `route_task_result/2` called with unknown `task_id` → log warning, return `{:error, :unknown_task}`.
  - `PendingTasks` ETS table not started → `route_task_result/2` returns `{:error, :pending_tasks_unavailable}`.
- **Delivery guarantees:** At-most-once for `TaskRequest` push (no retry at this layer — Provider.External owns retry/timeout semantics). Exactly-once for result delivery (PendingTasks entry is consumed on resolve).
- **Multi-tenancy:** None for MVP.

## APIs (as relevant to this role)

### TaskPush Module API

```elixir
# Push a TaskRequest to an agent via their registered transport.
@spec push(:grpc, pid(), map()) :: {:ok, :sent} | {:error, :transport_down | :send_failed}
@spec push(:websocket, pid(), map()) :: {:ok, :sent} | {:error, :transport_down | :send_failed}
def push(transport, pid, task_request)
```

- **gRPC path:** Sends `{:push_gateway_message, %GatewayMessage{msg: {:task_request, task_req}}}` to the stream pid (same pattern used by `GrpcServer` for direct messages and broadcasts).
- **WebSocket path:** Sends `{:push_to_agent, "task_request", payload}` to the channel pid (same pattern used by `GrpcServer.push_to_agent/4` for WebSocket delivery).
- **Error handling:** Checks `Process.alive?/1` before sending. Returns `{:error, :transport_down}` if pid is dead.

### Updated Registry.route_task_result/2

```elixir
# Route a completed task result to the waiting caller.
@spec route_task_result(String.t(), map()) :: :ok | {:error, :unknown_task}
def route_task_result(task_id, result)
```

- Calls `PendingTasks.resolve_task(task_id, result)` to deliver result to the blocked caller.
- On success: logs at debug level, emits `[:cortex, :gateway, :task, :completed]` telemetry.
- On `{:error, :not_found}`: logs warning ("unsolicited task result for task_id=..."), returns `{:error, :unknown_task}`.

### Updated AgentChannel.handle_task_result/2

Currently:
```elixir
defp handle_task_result(payload, socket) do
  case Protocol.validate_task_result(payload) do
    {:ok, _task_result_msg} ->
      {:reply, {:ok, %{}}, socket}
    ...
```

Updated to call `Registry.route_task_result/2` with the validated result:
```elixir
defp handle_task_result(payload, socket) do
  case Protocol.validate_task_result(payload) do
    {:ok, task_result_msg} ->
      result_map = %{
        "status" => task_result_msg.status,
        "result_text" => task_result_msg.result_text,
        "duration_ms" => task_result_msg.duration_ms,
        "input_tokens" => task_result_msg.input_tokens,
        "output_tokens" => task_result_msg.output_tokens
      }
      Registry.route_task_result(task_result_msg.task_id, result_map)
      {:reply, {:ok, %{}}, socket}
    ...
```

### How messages flow through these components

```
Provider.External.run/3
  ├─ registers pending task via PendingTasks.register/2
  ├─ finds agent via Registry.list/1
  ├─ gets push info via Registry.get_push_pid/2
  ├─ calls TaskPush.push(transport, pid, task_request)  ← I own this
  │     ├─ :grpc  → send({stream_pid, {:push_gateway_message, msg}})
  │     └─ :websocket → send({channel_pid, {:push_to_agent, "task_request", payload}})
  ├─ blocks on receive/GenServer.reply
  │
  │  ... agent works ...
  │
  ├─ gRPC path: GrpcServer receives TaskResult
  │     → calls Registry.route_task_result/2  ← I wire this
  │         → PendingTasks.resolve_task/2 → unblocks caller
  │
  └─ WebSocket path: AgentChannel receives "task_result"
        → calls Registry.route_task_result/2  ← I wire this
            → PendingTasks.resolve_task/2 → unblocks caller
```

### Concurrency model

- `TaskPush.push/3` is called from the Provider.External process (the caller's Task/process). It sends an async message to the stream/channel pid. No blocking, no GenServer call.
- `route_task_result/2` is called from the GrpcServer stream process or the AgentChannel process. It performs an ETS lookup and sends a message to the waiting caller. No serialization through Registry GenServer.

1. **Every `TaskRequest` pushed via `TaskPush.push/3` reaches the agent's transport process or returns an error** — no silent message loss.
2. **A `TaskResult` with a valid `task_id` in PendingTasks is delivered exactly once** — the ETS entry is consumed atomically by `PendingTasks.resolve_task/2`.

6. **Telemetry events are emitted for both dispatch and completion** — using the already-defined events in `Cortex.Telemetry`.

## Engineering Decisions & Tradeoffs

### Decision 1: `TaskPush` as a stateless module, not a GenServer

- **Decision:** `TaskPush` is a plain module with `push/3` as a pure function — no process, no state.
- **Alternatives considered:** GenServer that holds transport connections or tracks in-flight pushes.
- **Why:** The push operation is a fire-and-forget `send/2` to an existing process. There's no state to manage — the transport pid is looked up from Registry by the caller. Adding a GenServer would introduce unnecessary serialization and complexity.
- **Tradeoff acknowledged:** No built-in retry or queue. If the transport pid dies between the `Process.alive?/1` check and the `send/2`, the message is lost. This is acceptable because Provider.External owns timeout/retry semantics — it will detect the failure when no `TaskResult` arrives within the timeout.

### Decision 2: `route_task_result/2` as a module function, not a GenServer call

- **Decision:** Keep `route_task_result/2` as a module-level function that directly calls `PendingTasks.resolve_task/2` (ETS lookup), rather than routing through the Registry GenServer.
- **Alternatives considered:** Making it a `GenServer.call` to the Registry process, which would serialize result delivery through the registry.
- **Why:** The Registry GenServer manages agent registration state. Task result routing is a separate concern that doesn't need to read or modify agent state. Routing through the GenServer would create a bottleneck — every task result would serialize behind agent registration calls. ETS is designed for concurrent reads.
- **Tradeoff acknowledged:** `route_task_result/2` operates outside the Registry GenServer's state, so there's no transactional guarantee between agent state changes and result delivery. This is fine because the two operations are logically independent.

### Decision 3: Use `Process.alive?/1` pre-check in TaskPush

- **Decision:** Check `Process.alive?/1` before sending, return `{:error, :transport_down}` if dead.
- **Alternatives considered:** Just `send/2` and rely on the caller's timeout to detect failure; or use `Process.monitor/1` for reliable detection.
- **Why:** `Process.alive?/1` is cheap and catches the common case (agent disconnected, channel process terminated). The TOCTOU race between check and send is real but rare, and the caller (Provider.External) has its own timeout as a backstop.
- **Tradeoff acknowledged:** TOCTOU race — pid can die between `alive?` check and `send`. Accepted because the timeout mechanism in Provider.External handles this case.

## Risks & Mitigations

### Risk 1: PendingTasks module API not yet defined

- **Risk:** The External Provider Engineer owns `PendingTasks`. If its API differs from what `route_task_result/2` expects, integration breaks.
- **Impact:** Blocked on implementation until API is agreed.
- **Mitigation:** The system context specifies `PendingTasks.resolve_task/2` as the contract. Coordinate with External Provider Engineer before implementation to confirm: `resolve_task(task_id, result_map) :: :ok | {:error, :not_found}`.
- **Validation time:** 5 minutes (read their plan, confirm API shape).

### Risk 2: GatewayMessage proto struct not generated in Elixir

- **Risk:** The proto file defines `TaskRequest` in `GatewayMessage`, but if Elixir protobuf codegen hasn't been run, the struct `Cortex.Gateway.Proto.TaskRequest` may not exist.
- **Impact:** `TaskPush` can't build the gRPC message envelope.
- **Mitigation:** Check that `Cortex.Gateway.Proto.TaskRequest` and `Cortex.Gateway.Proto.GatewayMessage` modules exist (they're imported in `grpc_server.ex` already, so they should exist). If missing, run protobuf codegen.
- **Validation time:** 2 minutes (`mix compile` and check for the module).

### Risk 3: WebSocket Protocol.validate_task_result doesn't expose all fields

- **Risk:** `Protocol.validate_task_result/1` may not parse `task_id`, `result_text`, `input_tokens`, `output_tokens` from the payload — the current `handle_task_result/2` ignores the validated result.
- **Impact:** `route_task_result/2` receives incomplete data.
- **Mitigation:** Read `Protocol.validate_task_result/1` before implementation to confirm it extracts all needed fields. If it doesn't, update the Protocol validation (small, scoped change).
- **Validation time:** 5 minutes (read the module, check struct fields).

### Risk 4: `send/2` to gRPC stream pid uses wrong message format

- **Risk:** The gRPC stream process may not handle `{:push_gateway_message, msg}` for `TaskRequest` messages — it's only been tested with `DirectMessage` and `RegisterResponse`.
- **Impact:** TaskRequest messages are silently dropped by the stream process.
- **Mitigation:** Read `GrpcServer.connect/2` and trace how `{:push_gateway_message, msg}` is handled. The `GRPC.Server.send_reply/2` pattern should work for any `GatewayMessage` variant. Verify with a targeted test.
- **Validation time:** 10 minutes (read grpc_server.ex, write a test).

## EXPLAIN.md contributions (proposed outline bullets)

### Flow / Architecture
- `TaskPush.push/3` is the outbound half of the external provider dispatch — it takes a transport type and pid (from `Registry.get_push_pid/2`) and sends the `TaskRequest` via the appropriate protocol (gRPC `GatewayMessage` envelope or WebSocket channel event).
- `Registry.route_task_result/2` is the inbound half — called by both `GrpcServer` (gRPC path) and `AgentChannel` (WebSocket path) when a `TaskResult` arrives, it resolves the pending task and delivers the result to the blocked caller.
- The two halves are decoupled: `TaskPush` doesn't know about `PendingTasks`, and `route_task_result` doesn't know about transports. `Provider.External` composes them.

### Key Engineering Decisions
- `TaskPush` is stateless (no GenServer) because push is a fire-and-forget `send/2` — no state to manage.
- `route_task_result/2` accesses ETS directly instead of routing through the Registry GenServer, avoiding serialization on a per-task-result basis.
- `Process.alive?/1` pre-check in `TaskPush` catches the common case of dead transports; the TOCTOU race is covered by Provider.External's timeout.

### Limits of MVP
- No retry logic in `TaskPush` — if the transport pid dies between check and send, the message is lost. Provider.External's timeout handles this.
- No backpressure — `TaskPush` sends immediately regardless of agent load. Future: check `agent.load.queue_depth` before pushing.
- No transport-level acknowledgement — `TaskPush` returns `{:ok, :sent}` when the message is enqueued to the process mailbox, not when the agent actually receives it.
